# 通用 Agent Harness 开发指导文档

> 本文档基于对当前最佳实践的深度剖析，提取了与具体工具或特定的驱动开发模式（如 Spec-driven）解耦的通用设计哲学与设计范式。

## 1. 引言与设计哲学 (Design Philosophy)

优秀的 Agent Harness 绝不仅仅是一层将 LLM 包装成 CLI 命令的薄膜。它的本质是解决大语言模型在长期、复杂任务中的**上下文管理、自主权边界和验证可追溯性**问题。

开发一个强大的 Agent Harness，其核心设计理念应遵循：
- **以机器执行为中心，非以人读为中心**：Agent之间传递的各种中间态文档或参数，本质上都是 Prompt。这些文件需要包含高维的输入、输出、状态和指令，而不要混入冗余的为了让人类读懂而产生的企业级流程废话（如冗长的需求评审、项目矩阵）。
- **极简编排（Thin Orchestrator）原则**：Orchestrator（调度器/主框架）应该轻量，只负责验证规则、加载配置、调配资源、生成子智能体（Agent）。重活儿和具体的业务知识必须交给专精的 Subagent（如 Planner, Executor）。
- **可复用与模块化**：每一个环节（计划、执行、审查、验证）都是独立的上下文盒子，无论在什么时候插入一个新的任务，都不会导致系统崩溃。

---

## 2. 上下文工程 (Context Engineering)

LLM（大语言模型）最大的敌人是**上下文腐化 (Context Rot)**。给模型的上下文越多，其遵循核心指令的精确度越低。

在 Agent Harness 的设计中：
- **物理隔离**：摒弃长篇大论的“对话历史”，将历史压缩到**特定的结构化文件**中：
  - `ROADMAP/PROJECT/CONTEXT`：记录高层指导原则，确保“我在做什么/我为什么做”。
  - `STATE.md / PROGRESS`：记录“我已经做了什么，遇到了什么问题”。
  - `SUMMARY.md`：记录单次执行的精炼成果。每次切换上下文，老对话被丢弃，旧总结变新输入。
- **动态按需加载 (Discovery Protocol)**：Agent 不要在每次操作前都无脑读取整个代码库。对于已有的已知组件，直接检索模式使用；当需要做出架构选择或引入新的第三方依赖时，才触发“深度研究模式（Research）”，利用另外的智能体生成关于特定领域知识的总结之后，再给到执行环节。
- **守信质量降级曲线**：尽量将任意 Agent 的任务范围控制在消耗总上下文的 30% 到 50% 以内，避免让 LLM 处于“精疲力竭、开始敷衍”的疲惫状态。

## 3. 多智能体编排与隔离 (Multi-Agent Orchestration)

一个孤立的 Agent 无法处理中大型系统工程。在 Harness 中，我们需要实现**基于角色隔离**的工作流编排。

### 3.1 明确角色分工
- **Planner (规划者)**：不写代码，只进行任务拆解与逻辑链推演。构建“执行任务图”。
- **Executor (执行者)**：只有一件事：接收一份“无歧义的拆解任务”（XML结构），开始行动。
- **Verifier (校验者)**：以第三方的视角、根据 Planner 阶段设定的目标而非具体步骤，进行独立验证。

### 3.2 任务隔离与执行波次 (Wave Execution)
如果 Planner 将任务分成 A、B、C。在将任务传递给 Executor 时，必须实现：
- **依赖树与非冲突并发**：分析子任务的读写文件作用域 (File Ownership)。如果互相独立，Harness 应在不同波次（Wave）下并行启动多个 Executor 分区执行。
- **垂直切片优于水平切片**：例如开发带有界面、API 和数据模型的“用户管理模块”优先于“开发所有数据库模型然后开发所有API”。这允许端到端的全量测试，降低执行阶段对上下文的历史依赖。

## 4. 确权机制与自主性边界 (Autonomy & Deviations)

构建可用的 Harness 环境，不能奢求 Agent 一把梭不出错，更不能每次遇到一点异常都中断来问人类。我们要构建**弹性自主边界（Deviation Rules）**。

### 4.1 自动修复规则 (Auto-Fix)
如果 Agent 遇到异常问题：
- **Rule 1: Bug 修复** — 代码抛异常了、未通过静态分析？Harness 应允许 Agent 内部重试并修改（设置合理的上限，如重试 3 次）。
- **Rule 2: 补充关键前置** — 发现原定计划缺乏了某些错误处理或基本的安全约束（如验证模块忘加），在不改变架构的前提下，自动补全。
- **Rule 3: 解除阻塞** — 如果因缺少配置、网络等问题卡住，允许使用工具尝试修复。

### 4.2 设置检查点机制 (Checkpoints Protocol)
如果问题超出自主权，触发特定的检查点退出闭环，由 Orchestrator 阻塞进程，呼唤开发者：
- **决策检查点 (`decision`)**：需要大架构变更？抛给用户来选择备选方案。
- **验证检查点 (`human-verify`)**：视觉或者某些只能通过人类主观评价的体验测试。
- **动作检查点 (`human-action`)**：处理短信验证码、第三方外部OAuth登录等确需人为介入的动作。

## 5. 面向机器的通信规范 (Machine-Oriented Interop)

使用特定的**标记语言结构体（如 XML、YAML 等）**作为各环节 Agent 间传递协议，这是减少大语言模型“理解幻觉”的最佳方式。

### 5.1 明确规范的载体
每个分配给执行者 (Executor) 的 Plan 必须是明确且自动校验的。例如：
- `files`：指明必须且仅限修改哪些文件目录。
- `action`：带有确定性原则的实施方法（不含糊说“加个路由”）。
- `verify`：**必须提供一个不需要人参与的自动化验证命令（Bash 脚本或者测试命令）**。没有这个环节，Harness 系统就不算闭环。

### 5.2 “逆推溯源”法 (Goal-Backward Methodology)
如果计划系统要制定一个完整的执行协议（XML Prompt）：
- 它不能只是从实现角度正向想出 `A->B->C` 步骤，更需要先提出：**“要想确保功能完成，系统需要表现出哪些必然现象（Observable Truths）？”**
- 让 Agent 通过真实现象反推到所需的系统连接关键图谱（Key Links），最后根据关键图解列出验收命令。

---

## 6. 可观测性与系统状态管理 (Observability & State Management)

由于 Agent 在后台长线运行，开发者需要清晰地看到进展以及确保状态随时可追踪回滚。

### 6.1 原子化提交与历史溯源 (Atomic Commits)
Harness 应默认接管环境的 Git 操作，确保每一个细分阶段（如，某个 XML Task 成功）执行立刻进行单独 Commit 并标记类型 (`feat/test/fix`) 。
- 这样能随时让开发人员定位到底哪一个任务动作引入了 Bug。
- 在后续给 Agent 的上下文中，简短清晰的 Commit History 有助于降低模型的“认知成本”。

### 6.2 工作区总结法 (Summary Protocol)
在任意阶段结束前，Agent 必须创建用于归档本次动作的 `SUMMARY` 面板：
- 说明做了什么。
- 记下遇到了哪些不在原有指令中的问题（Deviations）。
- 跟踪是否有**埋下的存根（Stubs/Placeholders）**尚未处理——这个极为关键，避免后续任务在遇到空接头或伪造数据时无法继续推进。

---

## 结语

开发 Agent Harness，最关键的是构建**一套稳定的人机相互信任系统**：既不要让大语言模型陷入大海捞针（上下文腐化），又不能给它绝对自由而乱动基建（缺乏原子 Commit 与状态回溯）。使用结构化、层层隔离的手段，是当下开发大型通用 Harness 框架的不二法则。
